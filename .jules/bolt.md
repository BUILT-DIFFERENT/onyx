## 2024-05-16 - Fixed N+1 query in Note Search
**Learning:** Room DAOs can easily handle batch fetching using `IN (:ids)` lists, which drastically reduces database roundtrips. Inside `searchNotes` in `NoteRepository.kt`, mapping over each result and querying the database line-by-line led to massive N+1 slowdowns as the search hit list grew.
**Action:** Replaced O(N) database queries with O(1) batch queries using `getByIds` and mapped the results in memory using `associateBy`. Added an early return `if (recognitionHits.isEmpty()) return@map emptyList()` to avoid SQLite `IN ()` syntax errors. Next time, always check loops in repository layers for N+1 query patterns.
## 2024-05-18 - Avoid hypot() in Android Touch & Rendering Hot Loops
**Learning:** `kotlin.math.hypot()` internally calls Java's `Math.hypot()`, which performs several hidden mathematical checks for overflow, underflow, Infinity, and NaN handling. This makes it significantly slower than direct multiplication for small values that will never exceed precision limits (like screen coordinates or touch velocities). Inside 60Hz/120Hz loops, such as touch processing in `InkCanvasTransformTouch.kt` or `withFrameNanos` loops in `NoteEditorScreen.kt`, these conversions (`Double` back to `Float`) and hidden operations add up.
**Action:** Always replace `hypot()` inside Android hot loops with explicit squared distance logic (e.g., `x*x + y*y`), and compare against squared threshold constants. If the actual distance value is strictly needed, use `sqrt(x*x + y*y)`. This prevents Double conversion overhead and eliminates the unnecessary bounds checks.
